# 设计文档最小完备标准（可重建要求）

> 目标：确保 `docs/design/` 能独立指导实现与重建，不依赖隐式知识。

## 1. 适用范围

- `docs/design/Architecture.md`
- `docs/design/Interfaces.md`
- `docs/design/modules/**/README.md`
- 任何作为“当前标准”的设计文档

## 2. 强制章节（缺一不可）

每份设计文档必须显式包含以下 5 类内容：

1. 总体架构  
   - 模块定位、边界、依赖关系、约束。
2. 核心流程  
   - 主流程时序（输入 -> 处理 -> 输出），关键分支和状态迁移。
3. 数据结构  
   - 核心数据模型、关键字段、字段约束、序列化约定。
4. 关键接口  
   - 对外契约、参数/返回、调用方与被调用方责任。
5. 异常与错误处理  
   - 错误分类、重试/降级/回滚策略、可观测性与审计要求。

## 3. 推荐章节（建议补齐）

- 配置与开关语义
- 安全与信任边界
- 测试与验证策略
- 与其他模块的集成约束
- 已知限制与迁移计划

## 4. 编写规则

- 必须给出可定位证据：文档路径、代码路径、接口路径。
- 文档更新应与实现变更同 PR 提交，避免“文档滞后”。
- 若设计与实现不一致，先更新设计并记录 gap，再改代码。

## 5. 评审清单（最小）

- [ ] 5 个强制章节齐全且内容可执行  
- [ ] 关键流程覆盖成功路径和失败路径  
- [ ] 接口签名与代码一致（含异常语义）  
- [ ] 有明确的 gap 与 TODO 映射（若存在差异）  
- [ ] 文档可让新成员独立实现核心流程
