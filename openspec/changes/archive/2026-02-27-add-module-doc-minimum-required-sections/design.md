## Context

当前模块文档（`docs/design/modules/*/README.md`）大多已有“对外接口 / 关键字段 / 关键流程”结构，但未显式包含“总体架构”和“异常与错误处理”章节，无法满足 `Design_Doc_Minimum_Standard.md` 的 5 类强制内容要求。该缺口会导致文档评审标准不一致，也削弱“代码缺失时可重建”的目标。

本次变更是跨模块文档治理改造，范围覆盖所有模块 README。

## Goals / Non-Goals

**Goals:**
- 为模块 README 显式补齐缺失的“总体架构”“异常与错误处理”章节。
- 保留现有章节结构，采用统一补充块方式减少冲突与维护成本。
- 在补充章节中提供可定位证据（代码路径/接口路径），满足可追溯要求。

**Non-Goals:**
- 不重写各模块全文档。
- 不引入新的运行时能力或接口行为。
- 不替代 DG-001 的架构层对齐工作。

## Decisions

1. 采用统一“最小标准补充”章节模式。
   - 在每个模块 README 末尾增加 `总体架构` 与 `异常与错误处理` 子章节。
   - 通过“映射到已有章节 + 关键实现路径”避免大规模结构重排。

2. 保持模块差异化内容最小但可执行。
   - 每个模块补充块包含该模块的主要边界、依赖域、典型错误与降级策略。
   - 避免仅复制模板导致“形式完整但不可执行”。

3. 使用关键词回扫做验收。
   - 验收信号：每个 README 命中“总体架构/核心流程/数据结构(或关键字段)/关键接口(或对外接口)/异常与错误处理”。

## Risks / Trade-offs

- [Risk] 批量编辑导致编号不一致或冲突 → Mitigation: 新增独立补充章节，避免改动已有编号。
- [Risk] 补充内容过于笼统 → Mitigation: 每份补充至少给出 1-2 个模块内真实错误路径与处理策略。
- [Risk] 后续新增模块再度缺章 → Mitigation: 在 TODO 与 spec 中固化门禁要求，后续可接 CI 检查。

## Migration Plan

1. 扫描并确认目标 README 列表。
2. 批量追加“最小标准补充”章节。
3. 执行关键词回扫与 OpenSpec strict validate。
4. 回写 DG-007 状态与证据。
5. 同步 specs 并归档 change。
